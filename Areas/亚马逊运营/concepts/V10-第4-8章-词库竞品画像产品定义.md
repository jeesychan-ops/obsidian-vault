# V10 第4-8章：词库 → 竞品库 → 用户画像 → 产品Brief → 产品定义表

本章是 V10 框架中**从"市场分析"切换到"产品定义"的枢纽**。前面的市场/节点/商品集回答的是"做什么方向"，这里回答的是"具体做什么产品、怎么做"。

---

## 第4章：词库建设（Keyword Library）

### 4.1 词库的战略作用

词库不是"给Listing堆关键词"，而是**支撑四大模块的底层数据资产**：

| 消费模块 | 词库的作用 |
|---------|-----------|
| **Listing 撰写** | 标题/五点/A+ 的 SEO 骨架 |
| **广告架构** | Campaign 命名、关键词匹配、否定词 |
| **视觉方向** | 场景词 → 附图场景选择 |
| **否定投放** | 弱相关/无关词 → 保护利润 |

### 4.2 词库分层标准

词库必须分成 **5 层**，每层有不同的运营目的：

```
Layer 1: 品牌词（自家品牌 + 竞品品牌）
  → 用途：SB广告 / 品牌防御 / 品牌旗舰店承接
  → 示例：Anker、Sony、自家品牌名

Layer 2: 核心词（高搜索量、高转化意图）
  → 用途：标题必含、精准广告主力
  → 示例：wireless earbuds、bluetooth headphones
  → 判断标准：月搜索量 10,000+、CVR > 类目均值

Layer 3: 属性词（材质/功能/形态/尺寸）
  → 用途：五点描述、A+ 模块、否定投放
  → 示例：noise cancelling、waterproof、USB-C
  → 判断标准：搜索量 1,000-10,000、修饰核心词

Layer 4: 场景词（使用场景/人群/时机）
  → 用途：附图方向、A+ 场景模块、Rufus 问答
  → 示例：for commuting、gym workout、travel
  → 注意：这是 Rufus/COSMO 算法最看重的词类

Layer 5: 长尾词（低竞争、高转化、精准流量）
  → 用途：广告拓量、低 ACOS 流量源
  → 示例：wireless earbuds for small ears
  → 判断标准：搜索量 100-1,000、竞价低、转化稳定
```

### 4.3 工具链与采集流程

| 步骤 | 工具 | 产出 |
|------|------|------|
| 1. 种子词采集 | Sorftime 关键词挖掘 / 卖家精灵 / H10 | 初始词库（200-500个） |
| 2. 竞品反查 | 上述工具的反查功能 | 竞品流量词表 |
| 3. 搜索词报告 | 自动广告跑 7-14 天 | 真实用户搜索词 |
| 4. ABA 品牌分析 | 亚马逊官方 Search Query Performance | 点击/转化数据 |
| 5. 分类打标 | 按 4.2 分层标准归类 | 结构化的 5 层词库 |

### 4.4 词库管理的"铁律"

1. **永远按 ASIN 建库**——同一个类目的不同产品，词库不同。不能复用。
2. **每周更新一次**——搜索趋势在变，竞品在加词，词库必须活。
3. **否定词库同步维护**——跑出"买家的真实点击"≠"我们的目标词"，浪费预算的必须否定。
4. **场景词是 Rufus 时代的新金矿**——传统词库只关注"产品叫什么"，现在必须增加"在什么场景下被使用"。

## 第5章：竞品库建设（Competitor Library）

### 5.1 竞品库的核心目的

不是"看看对手长什么样"，而是**回答三个问题**：
1. **打谁**：锚定哪个/哪些竞品作为攻击目标？
2. **怎么打**：竞品的弱点在哪？我们的差异化入口在哪？
3. **守什么**：哪些竞品可能反击，怎么防守？

### 5.2 锚定竞品选择标准

不是所有同类产品都算竞品。按"价格带+用户任务"锁定 5-10 个：

| 竞品类型 | 定义 | 数量 | 用途 |
|---------|------|------|------|
| **锚定竞品** | 价格差 ±20%、用户画像高度重合 | 3-5 | 攻击目标、Listing 对标 |
| **价格锚点** | 价差 >30%、但在同一类目 | 2-3 | 定价参考、差异化证明 |
| **流量竞品** | 广告位/自然位常出现 | 2-3 | 广告防御、ASIN 定投 |

### 5.3 竞品分析八维表

每个锚定竞品必须完成以下八维扫描：

```md
┌──────────────────────────────────────────────────┐
│  竞品 ASIN：B0XXXXXXX                            │
│  品牌：品牌名  │  价格：$XX.XX  │  评分：4.X     │
├──────────────────────────────────────────────────┤
│  1. 流量入口：自然搜索词 Top 5 / 广告词 Top 5    │
│  2. 价格策略： Coupon / 多 variant / 套装打法    │
│  3. Review 分析：好评高频词 / 差评高频痛点       │
│  4. 视觉策略：主图风格 / 附图顺序 / 视频有无     │
│  5. A+ 模块：用了哪些模块 / 信息层次             │
│  6. 广告强度：自动/手动/ASIN定投 预估日均花费    │
│  7. BSR 趋势：最近 3 个月排名波动（上升/下降）   │
│  8. FBA 库存：库存健康度（In Stock 频率）        │
└──────────────────────────────────────────────────┘
```

**关键操作：** 差评分析是所有分析的起点。把竞品最近 1-3 个月的差评复制出来，按"产品问题/服务问题/期望问题"分类，这就是你的产品定义输入源。

### 5.4 竞品库的"攻击路径"推导

从八维表推导三条路径：

| 路径 | 适用场景 | 动作 |
|------|---------|------|
| **正面突破** | 竞品弱点是产品问题（材质差/功能不足） | 用更好的产品 + 差评黑转白文案 |
| **侧翼夹击** | 竞品弱点是价格（定价偏高） | 同类品质低 15-20% 定价 |
| **降维打击** | 竞品全优，但没覆盖某个使用场景 | 针对该场景重新定义产品 |

### 5.5 竞品库不可逆规则

- ❌ 不要包含售价相差 >50% 的产品（用户决策逻辑不同）
- ❌ 不要包含 BSR 排名 1-3 的超级头部（他们的流量逻辑是品牌驱动，不是搜索驱动）
- ✅ 锚定 BSR Top 10-50 的竞品，他们最容易被新品替代

## 第6章：用户画像与VOC表

### 6.1 为什么必须做画像

选品阶段如果跳过用户画像，后果是：**Listing 写出来不知道对谁说话，广告选词不知道买什么意图**。

用户画像回答三个问题：
1. **谁在买？**——人群特征
2. **为什么买？**——动机与场景
3. **为什么犹豫？**——疑虑与痛点

### 6.2 画像采集四渠道

| 渠道 | 获取什么 | 可信度 |
|------|---------|--------|
| **竞品差评** | 用户的真实不满、产品缺陷 | ⭐⭐⭐⭐⭐ |
| **竞品 QA** | 用户购买前的疑虑 | ⭐⭐⭐⭐⭐ |
| **竞品好评** | 用户最满意的点（放大） | ⭐⭐⭐⭐ |
| **VOC 报告** | 亚马逊官方用户反馈数据 | ⭐⭐⭐⭐⭐ |

### 6.3 VOC表模板

每个目标用户群体建立一条记录：

```yaml
用户画像 ID: USER-001
人群标签: 健身爱好者 / 年龄25-40 / 男性偏多 / 中高收入
使用场景: 健身房训练 / 户外跑步 / 通勤路上
购买动机: 运动时不希望有线缠绕 / 需要稳固定位
核心痛点: 现有产品运动时会脱落 / 音质在嘈杂环境不行
疑虑点:  防水等级不够 / 续航太短 / 价格偏高
原话证据:
  - "They keep falling off when I run" (差评)
  - "Is this waterproof for sweating?" (QA)
  - "Best for gym, not for everyday use" (好评)
```

### 6.4 画像到卖点的翻译规则

| 用户原话 | 翻译成卖点 |
|---------|-----------|
| "They keep falling off" | Secure-fit ear hooks design |
| "Battery dies too fast" | 12-hour playtime |
| "Sweat ruined them" | IPX5 waterproof rating |

**核心原则：用户语言 = 卖点语言。不要自己编词汇，从用户原话里取。**

## 第7章：产品 Brief

### 7.1 Brief 的核心定位

产品 Brief 是**从"市场研究"到"产品落地"的转换器**。

前面第4-6章（词库/竞品/画像）的输出 → 产品 Brief → 供应链/设计/文案/视觉团队执行。

### 7.2 Brief 完整内容清单

```
┌─────────────────────────────────────────────────┐
│              产品 Brief 模板                       │
├─────────────────────────────────────────────────┤
│ 1. 商品集结论（来自第3章商品集矩阵）                │
│    目标用户任务 / 核心机制 / 价格带                │
│                                                   │
│ 2. 目标用户画像（来自第6章VOC表）                  │
│    人群 / 场景 / 动机 / 痛点                      │
│                                                   │
│ 3. 产品形态定义（来自第8章）                      │
│    功能 / 材质 / 尺寸 / 颜色 / 套装               │
│                                                   │
│ 4. 差异化声明（来自第5章竞品分析）                 │
│    跟锚定竞品比，我们好在哪                        │
│                                                   │
│ 5. 规格参数表                                     │
│    重量 / 包装尺寸 / 材质 / 认证 / 合规           │
│                                                   │
│ 6. 定价区间（来自竞品价格带+成本测算）            │
│    安全线 / 利润线 / 广告容忍线                   │
│                                                   │
│ 7. 包装要求                                       │
│    FBA 合规包装 / 品牌 VI / 说明书语言            │
│                                                   │
│ 8. Listing 骨架                                    │
│    核心词 / 场景词 / 卖点优先级                    │
│                                                   │
│ 9. RFQ（采购询价单）                              │
│    数量 / 交期 / 质量要求 / 验货标准              │
│                                                   │
│ 10. Go / Pivot / Kill 条件                        │
│     什么状态下继续 / 转向 / 放弃                   │
└─────────────────────────────────────────────────┘
```

### 7.3 Brief 的核心原则

1. **拒绝模糊词**——"好材质"不算，必须写"医用级硅胶""7000系列铝合金"
2. **每个卖点必须有一个竞品弱点证据**——否则是自嗨
3. **RFQ 必须写"如果做不到就别报价"的底线**——供应链管理是产品管理的延伸
4. **Brief 写完不是结束，是团队对齐会议的输入**

## 第8章：产品定义表

### 8.1 产品定义表的作用

产品定义表是 Brief 的**可执行版本**——每个参数/属性/材质/颜色/尺寸都必须明确写出，直接给供应链、设计、文案、视觉团队使用。

没有产品定义表，就会出现：_文案写了"高端材质" → 供应链买的是普通材质 → 用户不满意 → 差评 → 退货。_

### 8.2 产品定义表模板

```yaml
产品名称: [产品名]
SKU 版本: [首个SKU / Variant A / Variant B]
目标价格: $XX.XX - $XX.XX

功能列表:
  - 功能1: [描述 + 技术参数]
  - 功能2: [描述 + 技术参数]
  - 功能3: [描述 + 技术参数]

材质清单:
  - 主体: [材质名] / [规格]
  - 配件: [材质名] / [规格]

物理参数:
  尺寸: XX x XX x XX cm
  重量: XXX g
  包装尺寸: XX x XX x XX cm
  包装重量: XXX g

颜色选项:
  - [颜色1] / [Pantone号]
  - [颜色2] / [Pantone号]

套装内容:
  - 本体 x 1
  - 配件1 x N
  - 说明书 x 1（语言: EN/FR/DE/IT/ES）

合规认证:
  - FCC / CE / RoHS / UL / FDA（按类目确认）
  - 儿童产品需 CPSIA

包装要求:
  - FBA 合规: 是/否
  - 是否需要展示包装（零售级）
  - 品牌 Logo 位置

印刷要求（说明书/背标）:
  - 语言
  - 必须包含的安全警示语

质保: XX 个月
```

### 8.3 产品定义的"六锚定"法则

每个产品定义必须锚定以下六个维度：

| 维度 | 每个产品都必须回答的问题 | 示例 |
|------|------------------------|------|
| **功能** | 它做什么？ | 无线充电、降噪、防水 |
| **属性** | 它是什么？ | 入耳式耳机、直立式吸尘器 |
| **材质** | 它用什么做的？ | 硅胶耳塞、铝合金外壳 |
| **形态** | 它长什么样？ | 圆形、方形、可折叠 |
| **颜色** | 它有什么颜色？ | 黑色/白色/军绿色 |
| **参数** | 它的数字是什么？ | 电池 500mAh、重量 50g |

**缺任何一锚定的产品定义 = 不完备的 Brief，不能进入下一阶段。**

### 8.4 产品定义表的"三方确认"门控

产品定义表填写完成后，必须经过三方确认才能进入第9章（定价/文案/视觉）：

```
□ 供应链确认：所有材质/尺寸/认证可批量生产，成本在预算内
□ 设计确认：颜色/形态/包装可执行，与品牌 VI 一致
□ 运营确认（你）：产品定义能支撑 Listing 卖点和广告词
```

三方签字后，产品定义表锁定。**后续任何修改必须走变更流程，不能随意改。**

## D-44 到 D-30 门控检查清单

| # | 检查项 | 输出物 | 通过标准 |
|---|--------|--------|---------|
| 1 | 词库闭环 | 结构化5层词库表 | 每层至少20个词，有搜索量/竞价数据 |
| 2 | 竞品库闭环 | 八维分析表 + 攻击路径 | 至少锁定3个锚定竞品，每条路径有证据支撑 |
| 3 | 画像闭环 | VOC 表 | 至少2个人群画像，有原话证据 |
| 4 | 产品 Brief 闭环 | 完整10项 Brief | 差异化声明有竞品证据对照 |
| 5 | 产品定义表闭环 | 六锚定定义表 | 缺一不可，三方确认已签字 |

**门控判断：**
- ✅ **Go** — 5项全部闭环，进入 D-29 定价/文案/视觉阶段
- ⏸️ **Pivot** — 3-4 项闭环，但产品定义不清晰 → 回炉第8章
- ❌ **Kill** — 词库/竞品/画像 任一环节做不出来或数据严重不匹配 → 回到第0章重选方向

## 新增：第12章 包装与说明书（来源：V10下篇预览）

> **适用阶段：D-30 到 D-7 | 前置依赖：第9-11章（定价/文案/视觉）已完成**
>
> 本章涵盖包装作为首图信息载体的规则、三件套一致性检查、说明书 8 模块 SOP、以及"先把差评写进说明书"的实战思路。

### 12.1 包装作为首图信息载体的规则（1600×1600 + 148×148 双验收）

#### 12.1.1 核心认知

包装不仅是物流保护工具，更是**首图的信息载体**。亚马逊移动端搜索页的图片尺寸逻辑决定了包装设计必须服从平台展示规则。

#### 12.1.2 1600×1600 首图验收标准

- **强制边长 1600×1600**：低于此标准的产品图片在移动端缩放后模糊，影响点击率
- **首图必须包含产品核心卖点标签**：当前行业趋势是首图上叠加 3-5 个卖点 icon（如 "12H Battery" "IPX5" "Secure Fit"），这些卖点信息直接来源于包装设计
- **包装主视觉 = 首图主视觉**：首图的产品展示角度、颜色、配件摆放方式必须与实物包装上印制的产品图一致

#### 12.1.3 148×148 缩略图验收标准

- **移动端搜索页缩略图强制 148×148**：首图核心信息的可读性必须在这个尺寸下验证
- **验收方法**：将 1600×1600 原图缩放到 148×148，检查：
  - 产品主体是否清晰可辨
  - 卖点文字是否仍可阅读（若不可读，说明卖点文字太大或太依赖小字）
  - 品牌 Logo 是否可识别
- **双验收流程**：设计稿出图后，必须先过 1600×1600 全尺寸审稿，再缩到 148×148 确认缩略图效果。**两个尺寸都通过才算验收通过**

#### 12.1.4 包装盒面信息层级

| 层级 | 内容 | 设计原则 |
|------|------|---------|
| 第一层 | 产品名称 + 产品图 | 148×148 缩略图下可见 |
| 第二层 | 核心卖点（3-5个） | 对应首图卖点 icon |
| 第三层 | 品牌 Logo | 统一 VI 位置 |
| 第四层 | 规格参数 / 合规标识 | FBA 标签预留位 |

---

### 12.2 页面/包装/说明书一致性检查

#### 12.2.1 一致性问题的代价

- **页面说 A，包装印 B，说明书写 C** → 用户认知混乱，退货率上升
- **实物与图片不一致** → 差评 "Not as described"，严重时触发亚马逊绩效审核
- **说明书功能描述与页面卖点不符** → 用户质疑产品质量，留下 1-2 星差评

#### 12.2.2 三件套一致性对照表

| 检查维度 | Listing 页面 | 包装盒 | 说明书 |
|---------|-------------|--------|--------|
| 产品名称 | 标题产品名 | 正面产品名 | 封面产品名 |
| 核心卖点 | 5点/Bullet Points | 正面卖点 icon | 功能介绍章节 |
| 颜色 | 主图颜色 | 盒子印刷颜色 | 产品图示颜色 |
| 规格参数 | A+ / Product Description | 侧面参数栏 | 规格表 |
| 配件清单 | A+ 配件展示 | 盒内清单 | 开箱清单 |
| 品牌名称 | Store / Title | Logo 位置 | 封面/封底 |

#### 12.2.3 一致性检查 SOP

1. **设计阶段（D-30）**：页面设计稿 + 包装设计稿 + 说明书排版稿 三稿对齐会议
2. **打样阶段（D-15）**：拿到实物打样后，对照 Listing 草稿逐项核查
3. **大货前（D-7）**：印刷厂出最终确认稿，运营/设计/供应链三方签字
4. **上架后（D+7）**：用户收到第一批货后，检查 QA 和首评是否出现 "与描述不符"

### 12.3 说明书 SOP（8 模块）

#### 12.3.1 说明书的设计目标

说明书是**用户开箱后的第一手体验资料**，直接影响开箱体验评分、首次使用成功率、退换货率和差评率。

#### 12.3.2 说明书 8 大模块

| 模块 | 内容 | 关键要点 |
|------|------|---------|
| **模块 1: 封面** | 产品名称/型号/SKU/品牌 Logo/"Quick Start Guide"/产品实物图 | 名称与 Listing 标题一致，产品图与首图一致 |
| **模块 2: 开箱清单** | 盒内所有物品图示+数量，配件名称（中英双语） | 明确标注不含的配件，减少"配件缺失"误会 |
| **模块 3: 产品图示** | 多角度标注图，按键/接口/指示灯功能标注，尺寸标注 | 解决"Much smaller than expected"尺寸预期错位 |
| **模块 4: 快速上手指南** | 从开箱到使用的 3-5 步极简流程，每步配图+一句话说明 | 解决"无法使用"类差评，降低退货率 |
| **模块 5: 功能介绍** | 每个功能+图示，对应 Listing 卖点，使用注意事项 | 明确标注**不支持**的功能，避免功能预期错位 |
| **模块 6: 规格参数表** | 尺寸/重量/材质/电池容量/认证标识/工作环境 | 与 A+ 参数栏完全一致 |
| **模块 7: 故障排除（FAQ）** | 用户最可能遇到的 5-10 个问题+解决方案 | 从竞品差评和 QA 中提取，直接引用差评原话 |
| **模块 8: 安全警告+保修** | 安全使用警告、保修条款、合规声明、售后服务 | 减少安全担忧类差评 |

#### 12.3.3 说明书质量控制

- **语言**：目标市场官方语言，美式英语拼写（Color 不是 Colour），避免机器翻译腔
- **纸张**：不低于 120g 铜版纸
- **印刷精度**：300 DPI 以上
- **色彩校准**：与包装盒 CMYK 色值一致

### 12.4 先把差评写进说明书

#### 12.4.1 核心理念

**"把差评消灭在说明书里"**——与其等到用户遇到问题给差评，不如在说明书中预先解答用户可能产生的困惑。

#### 12.4.2 差评→说明书映射表

| 差评类型 | 用户原话 | 说明书对应模块 |
|---------|---------|--------------|
| 使用困难 | "No instructions, couldn't figure out how to use it" | 模块 4 快速上手指南 |
| 功能不符预期 | "Thought it had XX feature but it doesn't" | 模块 5 功能介绍（标注不支持的功能） |
| 配对连接问题 | "Won't connect to my phone" | 模块 4 配对方法 + 模块 7 故障排除 |
| 配件缺失误会 | "Missing part, thought it came with Y" | 模块 2 开箱清单（标注不含的配件） |
| 尺寸/容量预期错位 | "Much smaller than expected" | 模块 3 尺寸标注 + 模块 6 规格参数 |
| 安全担忧 | "Is it safe to charge overnight?" | 模块 8 安全警告 |
| 兼容性问题 | "Doesn't work with Samsung" | 模块 5 兼容性说明 + 模块 7 故障排除 |

#### 12.4.3 实操方法

1. **竞品差评采集**：锚定竞品最近 3 个月的差评全部导出
2. **按问题类型分组**：使用困难/功能缺失/配件问题/尺寸/兼容性/品质/包装
3. **判断说明书能否解决**：能用说明书解决的写入对应模块；不能的必须在产品定义阶段解决
4. **FAQ 模块直接引用差评原话**：即使买家没看说明书，QA 区已有现成回答

#### 12.4.4 差评前置的优势

- 降低差评率：预期错位的用户在说明书阅读阶段调整预期
- 减少退货：因"不知道怎么用"而退货的比例显著下降
- 提升客服效率：FAQ 覆盖 80% 常见问题
- 改善开箱体验：用户从"困惑"变成"被引导"
